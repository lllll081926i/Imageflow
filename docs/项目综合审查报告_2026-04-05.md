## 一、CRITICAL 级别问题

### 1.2 `CancelActiveTask` 杀死整个 Python 进程而非单个任务
- **文件:** `backend/utils/python_executor.go:167-175`
- **描述:** 当前取消仍通过杀进程实现，虽然已避免误伤空闲 executor，且取消后会立即尝试重启 worker，但仍不是“仅取消当前任务”的优雅方案。
- **建议:** 将取消改为任务级协议，而不是进程级终止。

## 二、HIGH 级别问题

### 2.1 Go 后端

#### 2.1.1 `executeBatch` 取消后 goroutine 泄漏
- **文件:** `backend/app.go:588-637`
- **描述:** 取消请求后，主循环退出但已在执行的 worker 无法被取消。慢速 Python 执行会持续运行，goroutine 无法回收。
- **建议:** 实现上下文传递机制，使正在执行的任务也能响应取消。

#### 2.1.4 `SaveSettings` 状态不一致窗口
- **文件:** `backend/app.go:375-429`
- **描述:** 设置保存和内存状态更新之间存在时间窗口，并发 `Convert` 调用可能使用旧 runner 与新设置值。
- **建议:** 在单一锁保护下原子地完成设置保存和状态更新。

#### 2.1.5 `stopRunnerWhenIdle` goroutine 可能永久阻塞
- **文件:** `backend/app.go:316-330`
- **描述:** 如果 `operationsCond.Broadcast()` 从未被调用（如 panic 或遗漏信号），goroutine 将永远等待。
- **建议:** 添加超时机制或使用 context 控制生命周期。

#### 2.1.9 `app.go` 错误处理代码大量重复
- **文件:** `backend/app.go` 几乎每个方法
- **描述:** 每个方法都重复约 15 行的错误处理样板代码（~200 行重复）。
- **建议:** 提取为通用辅助函数。

#### 2.1.10 `App` 结构体违反单一职责原则
- **文件:** `backend/app.go:25-58`
- **描述:** `App` 包含 9 个服务、5 个互斥锁、1 个条件变量、预览缓存等，是典型的 God Object。
- **建议:** 拆分为 `PreviewManager`、`OperationTracker`、`ServiceRegistry` 等独立组件。

### 2.2 React 前端

#### 2.2.1 `DetailView.tsx` 组件过大（5000+ 行）
- **文件:** `frontend/components/DetailView.tsx`
- **描述:** 单文件 5031 行，约 60 个 `useState`，大量 `useMemo`/`useCallback`。即使使用 `memo()`，任何状态变化都会触发所有 useMemo 重新计算。
- **建议:** 拆分为功能组件（ConverterView、CompressorView、WatermarkView 等）或使用状态管理库（Zustand）。

#### 2.2.2 `handleStartProcessing` 缺少 `useCallback`
- **文件:** `frontend/components/DetailView.tsx:3270`
- **描述:** 捕获 60+ 状态变量的函数每次渲染都创建新实例，导致子组件不必要的重渲染。
- **建议:** 使用 `useCallback` 包裹或重构为 ref-based 命令模式。

#### 2.2.5 大量 `any` 类型
- **文件:** `frontend/components/DetailView.tsx` 多处
- **描述:** `FeaturePreset.payload: Record<string, any>`、`info: any | null`、`generateError: any` 等。
- **建议:** 定义完整的接口类型，使用 `unknown` 替代 `any`。

#### 2.2.6 渲染函数定义在组件内部
- **文件:** `frontend/components/DetailView.tsx:2173, 4512, 4636`
- **描述:** `renderSettings`、`renderInputSection`、`renderActionSection` 不是组件而是渲染函数，无法被 memo 优化。
- **建议:** 提取为带明确 props 的独立组件。

#### 2.2.9 Tailwind v4 使用遗留配置文件
- **文件:** `frontend/tailwind.config.js`
- **描述:** Tailwind v4 使用 CSS-first 配置（`@theme` 指令），`tailwind.config.js` 无效。
- **建议:** 迁移主题配置到 CSS `@theme` 块或删除该文件。

### 2.3 Python 脚本

#### 2.3.1 `_add_noise_slow` 和 Vignette 慢路径性能极差
- **文件:** `backend/python/filter.py:477-493, 519-539`
- **描述:** O(n²) 像素级 Python 循环。4K 图像需要数分钟。
- **建议:** 使用 numpy 数组操作或直接移除慢路径。

#### 2.3.2 `gif_splitter.py` 加载所有帧到内存
- **文件:** `backend/python/gif_splitter.py:453-470`
- **描述:** 大型动画 GIF（1000 帧 @ 1920x1080）将消耗数 GB 内存。
- **建议:** 实现流式处理或分块加载。

#### 2.3.3 `compressor.py` 无损模式读取整个文件到内存
- **文件:** `backend/python/compressor.py:346-353`
- **描述:** 超大 TIFF/JPEG 文件会导致 OOM。
- **建议:** 使用流式或分块处理。

#### 2.3.4 `pdf_generator.py` 同时加载所有图片到内存
- **文件:** `backend/python/pdf_generator.py:182-198`
- **描述:** 大批量高分辨率图片导致双倍内存压力。
- **建议:** 验证后释放句柄，按需加载。

#### 2.3.5 `subtitle_stitcher.py` 画布高度可能溢出
- **文件:** `backend/python/subtitle_stitcher.py:190`
- **描述:** 拼接数百帧时，画布可能超过 PIL 最大图像尺寸（~1.79 亿像素）。
- **建议:** 添加尺寸检查。

#### 2.3.6 `worker.py` 无超时或命令数限制
- **文件:** `backend/python/worker.py:140`
- **描述:** `while True` 循环无超时机制、无命令数限制、无内存监控。
- **建议:** 添加心跳检测、超时和最大命令数。

#### 2.3.7 测试覆盖率严重不足
- **文件:** `backend/python/tests/`
- **描述:** 缺少 `adjuster.py`、`converter.py`、`pdf_generator.py`、`worker.py` 的测试文件。
- **建议:** 补充核心模块的测试用例。

- **文件:** `.github/workflows/ci.yml`
- **描述:** CI 构建产物但不创建 GitHub Release，无 checksum、无签名。

#### 2.4.5 版本号不一致

- **描述:** `wails.json` 和 `package.json` 为 `1.0.10`，`pyproject.toml` 为 `0.0.0`。
