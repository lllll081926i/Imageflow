## 一、CRITICAL 级别问题

### 1.1 任意模块导入（无白名单）
- **文件:** `backend/engines/worker.py:162-165`
- **描述:** `importlib.import_module(module_name)` 直接使用来自 stdin JSON 的 `script_name`，攻击者可传入 `"os"`、`"subprocess"` 等触发模块级副作用。无允许的模块名白名单。
- **建议:** 维护允许的引擎模块名白名单，拒绝不在列表中的模块。

### 1.2 绝对路径绕过路径遍历检查
- **文件:** `backend/engines/worker.py:33-52` + `backend/domain/paths.py:21-26`
- **描述:** `_has_leading_parent_traversal` 对绝对路径直接返回 `False`，无目录沙箱限制。应用可读写任意文件。
- **建议:** 增加允许目录白名单校验，所有文件操作必须在已知应用目录内。

### 1.3 `IMAGEFLOW_SETTINGS_FILE` 环境变量无验证
- **文件:** `backend/infrastructure/settings_store.py:98-101`
- **描述:** 环境变量直接用作文件路径，`save_settings` 甚至会 `mkdir` 中间目录。可实现任意文件写入。
- **建议:** 校验路径必须在已知应用数据目录下。

### 1.4 TypeScript 未启用 strict 模式
- **文件:** `frontend/tsconfig.json`
- **描述:** 缺少 `strict: true`，整个前端无 `strictNullChecks`、`noImplicitAny`，所有 `as any` 转型无编译期报错。
- **建议:** 启用 `strict: true`，逐步修复类型错误。

### 1.5 SVG XML 解析存在 XXE 风险
- **文件:** `backend/engines/converter.py:113-127`
- **描述:** `ET.fromstring(data)` 只检查了 `<!DOCTYPE>` 和 `<!ENTITY>`，内部实体扩展攻击仍可能。
- **建议:** 使用 `defusedxml` 替代 stdlib ElementTree。

## 二、HIGH 级别问题

### 2.1 `resolve_file_paths` 遇到不可解析文件时丢弃所有结果
- **文件:** `backend/api/desktop_api.py:324-331`
- **描述:** `return []` 应改为 `continue`，当前逻辑：10 个文件第 7 个无法解析 → 全部丢失。
- **建议:** 将 `return []` 改为 `continue` 跳过不可解析的文件。

### 2.2 `get_image_preview` 缺少 `input_path` 键校验
- **文件:** `backend/api/desktop_api.py:208-210`
- **描述:** 裸字典访问 `normalized["input_path"]` 会抛 `KeyError`。
- **建议:** 添加 `get("input_path")` 或显式校验。

### 2.3 JS 注入风险：`evaluate_js` 使用字符串拼接
- **文件:** `backend/host/window.py:30-38`
- **描述:** `json.dumps(payload, ensure_ascii=False)` 直接拼入 JS 模板。
- **建议:** 改为 `ensure_ascii=True` 或传参方式。

### 2.4 `process.join()` 无超时 — 可永久阻塞
- **文件:** `backend/application/image_ops.py:58-59, 138-139` + `backend/application/task_manager.py:33-38`
- **描述:** 3 处 `process.join()` 缺少 timeout，进程不响应 `terminate()` 时应用挂死。
- **建议:** 统一添加 `join(timeout=N)` + 超时后 `kill()`。

### 2.5 批量处理 Worker 启动失败泄漏所有已启动进程和队列
- **文件:** `backend/application/image_ops.py:97-103`
- **描述:** `process.start()` 在第 N 个 worker 失败时，前 N-1 个 worker 和两个 Queue 无人清理。
- **建议:** 包装 worker 启动循环，失败时清理已启动的 worker。

### 2.6 Queue 从未关闭 — 文件描述符泄漏
- **文件:** `backend/application/image_ops.py:49-70`
- **描述:** 每次 `execute_engine` 调用泄漏一个 pipe FD 和一个 feeder 线程。
- **建议:** 在 `finally` 块中调用 `queue.close()` + `queue.join_thread()`。

### 2.7 非原子设置文件写入 — 断电即损坏
- **文件:** `backend/infrastructure/settings_store.py:125-130`
- **描述:** `write_text()` 非原子，中断后设置文件损坏且静默回退默认值，用户配置永久丢失。
- **建议:** 写入临时文件后 `os.replace()` 原子替换。

### 2.8 `sys.path.insert(0, ...)` 允许模块劫持
- **文件:** `backend/infrastructure/engine_loader.py:8-13`
- **描述:** `backend/engines/` 下的模块优先级高于标准库，存在模块遮蔽风险。
- **建议:** 改用 `sys.path.append()` 或 `importlib.util.spec_from_file_location()`。

### 2.9 Inkscape 临时文件在失败时不清理
- **文件:** `backend/engines/converter.py:320-344`
- **描述:** `subprocess.run` 异常时 `tmp_path` 永远残留在磁盘。
- **建议:** 用 `try/finally` 确保清理。

### 2.10 PIL Image 链式处理大量内存泄漏
- **文件:**
  - `backend/engines/adjuster.py:67-76` — 8 个中间 Image 未 close，4000x3000 RGBA 泄漏 ~384MB
  - `backend/engines/watermark.py:133-205` — overlay/watermarked/text_img 未 close
  - `backend/engines/filter.py:87-88` — RGBA 转 RGB 丢弃 alpha 通道
  - `backend/application/preview.py:39-43` — convert/copy 后原始 Image 泄漏
- **建议:** 统一 PIL Image 生命周期管理，每步操作后 close 旧 Image。

### 2.11 极慢的逐像素处理回退函数
- **文件:**
  - `backend/engines/filter.py:499-515` — 噪声：4000x3000 图 1200 万次 Python 循环
  - `backend/engines/filter.py:541-561` — 暗角：2500 次 draw.ellipse 循环
- **建议:** 使用 numpy 或 PIL 内置方法替代。

### 2.12 大文件全量读入内存
- **文件:** `backend/engines/gif_splitter.py:496-498, 521-523`
- **描述:** WebP/APNG 时长解析 `f.read()` 读取整个文件，数百 MB 文件可触发 MemoryError。
- **建议:** 使用分块读取 + `f.seek()` 只读取必要的 chunk。

### 2.13 CI Actions 未锁定 SHA
- **文件:** `.github/workflows/ci.yml:20,23,30,35,86,97`
- **描述:** 所有 `uses: actions/xxx@v4` 使用可变标签，供应链攻击风险。
- **建议:** 锁定到完整 SHA。

### 2.14 CI `contents: write` 权限覆盖所有事件
- **文件:** `.github/workflows/ci.yml:16-17`
- **描述:** PR 触发时也有写权限，恶意 fork PR 可利用。
- **建议:** `contents: write` 仅在 release 步骤启用。

### 2.15 `DetailView.tsx` 5000 行巨型组件
- **文件:** `frontend/components/DetailView.tsx`
- **描述:** ~70 个 useState，所有功能（转换/压缩/PDF/GIF/水印/调色/滤镜/信息）集中在一个组件。
- **建议:** 按功能拆分为独立组件 + 自定义 hooks。

### 2.16 前端无 ErrorBoundary
- **文件:** `frontend/App.tsx`
- **描述:** 任何渲染错误直接白屏，无恢复路径。
- **建议:** 在关键区域添加 ErrorBoundary。

### 2.17 `backend-models.ts` 全部构造函数使用 `any`
- **文件:** `frontend/types/backend-models.ts`
- **描述:** ~846 行自动生成代码，所有 `source: any = {}`，字段缺失时静默为 `undefined`。
- **建议:** 添加类型守卫或运行时校验。

## 五、架构层面问题

1. **路径安全缺乏统一策略：** `worker.py`、`paths.py`、`settings_store.py` 各自实现路径校验，逻辑不一致且均存在绕过。应抽取统一的 `PathValidator` 类。

2. **进程生命周期管理分散：** `image_ops.py`、`task_manager.py` 各自管理子进程，缺少统一的 `ProcessPool` 抽象，导致超时、清理、资源泄漏等问题反复出现。

3. **PIL Image 生命周期无规范：** 多个 engine 文件中 Image 对象的 create/convert/close 无一致模式，应引入 context manager 或统一的 `ImagePipeline` 类。

4. **前端类型系统形同虚设：** `backend-models.ts` 全 `any` 构造 + `tsconfig.json` 无 strict + 组件内大量 `as any`，TypeScript 退化为 JavaScript。

5. **`DetailView.tsx` 违反单一职责：** 8 个功能模块的状态和 UI 全部集中在一个文件，应按功能拆分为独立组件 + 自定义 hooks。

6. **错误处理策略不一致：** 部分函数抛异常、部分返回默认值、部分静默跳过，应建立统一的错误处理约定。
